Introduction to SSA

In compiler design, Static Single Assignment (SSA) form is a property of an intermediate representation (IR) that requires each variable to be assigned exactly once, and every variable to be defined before it is used. Dataflow analysis in traditional Abstract Syntax Trees (ASTs) often requires "reaching definition" calculations which scale quadratically, or O(N2). SSA reduces this overhead significantly.

Consider a simple block of code that computes a value and then overwrites it. In a standard three-address code structure, this mutation requires generating Kill and Gen sets for every basic block during liveness analysis. In a traditional AST, dataflow analysis becomes complicated because the definition of x changes mid-block. We can see the problem the inefficiency clearly below: 

// Original Code
int y = 1;
int x = 2;
x = x + y;
y = x * 2;

// SSA Form Translation
int y_1 = 1;
int x_1 = 2;
int x_2 = x_1 + y_1;
int y_2 = x_2 * 2;

To convert this to SSA form, we simply append unique subscripts to the variables. This gives us the mathematical property of referential transparency within the local block.

Compiler Phases

Before a JIT compiler applies SSA transformations, the raw source code must undergo several foundational phases. To execute this, type make build-jit in your terminal. A typical modern compiler is broken down sequentially into the following ordered steps:

  1. Lexical Analysis: Converts the raw string of code into a stream of categorized tokens. Output looks like [Token::IDENTIFIER("x")].
  2. Syntax Analysis (Parsing): Organizes the tokens into an Abstract Syntax Tree based on the language grammar.
  3. Semantic Analysis: Verifies type safety, variable resolution, and scoping rules.
"Speculative optimization is a gamble. We bet that the code will continue to execute with the same types it saw in the past. When we lose the bet, we must gracefully deoptimize."

In addition to these static phases, a JIT compiler applies speculative, runtime optimizations. These usually include:

  • Inline caching for polymorphic call sites to speed up dynamic dispatch.
  • Hidden class transitions (often implemented via Shape trees).
  • On-stack replacement (OSR) to compile hot loops while they are actively running.
View the OSR Lowering Algorithm

On-Stack Replacement is tricky because you must map the interpreter's virtual registers to the physical registers of the compiled machine code while the function is executing.

function OSR_Lower(frame_state, compiled_code):
    registers = MapVirtualToPhysical(frame_state)
    stack_offsets = ComputeStackSpills(frame_state)
    jump_to(compiled_code.loop_header, registers, stack_offsets)
